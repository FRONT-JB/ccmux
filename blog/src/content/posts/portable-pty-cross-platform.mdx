ccmux는 어떤 OS에서든 같은 2MB 바이너리 하나. Windows에서도 WSL 없이 동작합니다.

하지만 실은, **셸과 대화하는 仕組み는 OS마다 전혀 다릅니다**. 그 간극을 무엇으로 메우는지 이야기합니다.

## "셸과 대화하기"란

`bash`나 `zsh`는 상대가 **사람의 터미널**이 아니라고 감지하면 동작을 바꿉니다:

- 프롬프트 (`$ `)를 출력하지 않는다
- 색을 생략한다
- 줄 단위 입출력을 포기하고 전부 묶어버린다

파이프 (`ls | grep foo`)를 쓰면 색이 사라지는 경험 있으시죠? 그건 bash가 "상대는 터미널이 아니라 파이프구나"라고 알아채고 출력 형식을 바꾸기 때문입니다.

ccmux 같은 도구가 셸을 내부에서 실행할 때, 셸이 그걸 눈치채지 못하도록 **"아니, 제대로 터미널에서 실행 중이야"라고 믿게 만드는** 것이 필요합니다. 그를 위한 仕組みが **PTY**（pseudo-terminal, 가상 단말기）입니다.

## OS별 실제 구현

PTY의 구현은 OS마다 다릅니다:

| OS | 구현 |
|---|---|
| Linux | `openpty()` + `fork()` + `setsid()` |
| macOS | 유사한 Unix 계열 API |
| Windows 10+ | ConPTY（`CreatePseudoConsole`） |

ConPTY는 2018년 Windows 10 1809에서 추가된 비교적 새로운 仕組み입니다. 그 이전에는 WinPTY라는 서드파티 구현이 필요했고, 이것의 등장이 "Windows에서도 tmux 같은 것을 만들 수 있다"는 시대를 열었습니다.

## portable-pty가 전부 해결해준다

Rust에는 [**portable-pty**](https://crates.io/crates/portable-pty)라는 크레이트가 있어서 OS별 차이를 흡수해줍니다. wezterm（Rust로 만든 터미널 에뮬레이터）의 작자가 만든 것으로, 완성도가 높습니다.

같은 API로, 내부에서는 OS마다 다른 것이 동작합니다:

```rust title="src/pane.rs"
use portable_pty::{native_pty_system, CommandBuilder, PtySize};

fn spawn_shell(rows: u16, cols: u16) -> anyhow::Result<...> {
    let pty = native_pty_system(); // ← OS 자동 감지
    let pair = pty.openpty(PtySize { rows, cols, pixel_width: 0, pixel_height: 0 })?;

    let shell = detect_shell();
    let mut cmd = CommandBuilder::new(&shell);
    cmd.env("TERM", "xterm-256color");
    cmd.env("CCMUX", "1"); // ← 나중에 설명

    let child = pair.slave.spawn_command(cmd)?;
    drop(pair.slave); // slave는 자식 프로세스가 갖는다. 부모에서는 닫는다.

    let writer = pair.master.take_writer()?;
    let reader = pair.master.try_clone_reader()?;

    // ...
}
```

`#[cfg(windows)]`도 `#[cfg(unix)]`도 단 한 번도 쓰지 않았습니다. 이게 크로스 플랫폼의 쾌감입니다.

## 어떤 셸을 실행하는가

ccmux는 OS별로 셸 우선순위를 다르게 합니다.

**Windows:**

1. Git Bash (`C:\Program Files\Git\bin\bash.exe`) 최우선
2. PATH 상의 `bash`
3. 최후 수단으로 PowerShell

Windows에서 **Git Bash를 우선하는 이유**는 Claude Code를 포함한 많은 Unix 계열 CLI가 bash를 전제로 만들어져 있기 때문입니다. PowerShell에서는 `ls -la`나 `grep`이 그대로 통하지 않는 경우가 많아 불편합니다.

**Unix 계열（macOS / Linux）:**

1. `$SHELL` 환경변수（사용자의 평소 셸）
2. 없으면 `/bin/sh`

## 셸에 "지금 어디 있나"를 묻는 소기법

셸에서 `cd`를 해도, ccmux에서는 그 이벤트가 보이지 않습니다. 셸은 조용히 자신의 내부 상태만 바꿀 뿐입니다.

그래서 **프롬프트가 나올 때마다 셸 자체가 현재 경로를 알려주도록** 설정합니다:

```rust
let setup = " __ccmux_osc7() { \
    printf '\\033]7;file://%s%s\\007' \"$HOSTNAME\" \"$PWD\"; \
}; \
PROMPT_COMMAND=\"__ccmux_osc7;${PROMPT_COMMAND}\"; \
clear\n";
writer.write_all(setup.as_bytes())?;
```

`\033]7;file://...\007`는 **OSC 7**이라는 특수 이스케이프 시퀀스입니다. "지금 있는 디렉터리는 여기"라고 터미널에 전달하는 암호입니다.

ccmux는 이것을 vt100 파서로 받아서 페인별 cwd를 최신으로 유지합니다. 파일 트리나 새 페인의 시작 디렉터리가 **자동으로 "지금 있는 곳"**을 따라가는 것은 이 덕분입니다.

> 참고로 setup 앞에 **공백**을 넣는 것은, bash의 `HISTCONTROL=ignorespace`가 활성화된 환경에서 히스토리를 오염시키지 않기 위함입니다. 이런 세심한 배려가 빛을 발합니다.

## 중첩 실행 거부

ccmux 안에서 다시 ccmux를 실행하면 PTY 안에서 PTY가 돌아 입력이 이중 처리되어 **키 입력이 망가져 동작하지 않게 됩니다**.

이를 방지하기 위해 자식 셸에 `CCMUX=1`이라는 환경변수를 심어두고, ccmux 자체가 기동 시에 확인합니다:

```rust title="src/main.rs"
if std::env::var("CCMUX").is_ok() {
    eprintln!("ccmux: already running inside a ccmux pane.");
    std::process::exit(1);
}
```

"**왜인지 동작하지 않는다**" 类の버그를 사전에 차단할 수 있는, 소박하지만 가치 있는 한 줄입니다.

---

크로스 플랫폼 PTY는 직접 작성하려 하면 `#[cfg(...)]`가 대량으로 나와 힘들지만, **portable-pty에 맡기면 전혀 나오지 않습니다**. ccmux가 "Windows에서도 동작하는" 것은 작은 나름에도 이 크레이트의 혜택이 큽니다.
